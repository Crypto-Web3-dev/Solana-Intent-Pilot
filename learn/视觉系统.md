### 1. Rust 核心逻辑实现

首先，在你的 `core-engine` 目录下编写 `src/lib.rs`。这里我们直接解析 SPL Token 的 `Mint` 结构体。

Rust

```
use wasm_bindgen::prelude::*;
use solana_program::program_pack::Pack;
use spl_token::state::Mint;

// 定义返回给 JS 的风险报告结构
#[wasm_bindgen]
pub struct SecurityReport {
    pub is_rug_risk: bool,
    pub score: u8, // 0-100
    pub has_mint_authority: bool,
    pub has_freeze_authority: bool,
}

#[wasm_bindgen]
impl SecurityReport {
    #[wasm_bindgen(getter)]
    pub fn message(&self) -> String {
        if self.is_rug_risk {
            "Warning: High risk detected! Authority not renounced.".into()
        } else {
            "Safe: Ownership renounced and freeze disabled.".into()
        }
    }
}

/// 核心函数：解析 Mint 账户原始数据
#[wasm_bindgen]
pub fn analyze_mint_data(data: &[u8]) -> Result<SecurityReport, JsValue> {
    // 1. 利用 SPL Token 库将字节流反序列化为 Mint 结构体
    // 这比 JS 处理 Buffer 快得多且更安全
    let mint = Mint::unpack(data)
        .map_err(|_| JsValue::from_str("Failed to unpack Mint data"))?;

    let mut score = 100;
    let mut has_mint_auth = false;
    let mut has_freeze_auth = false;

    // 2. 核心风险检测逻辑
    // 如果 Mint Authority 存在，说明开发者可以随时增发代币（Rug 风险）
    if mint.mint_authority.is_some() {
        score -= 60;
        has_mint_auth = true;
    }

    // 如果 Freeze Authority 存在，说明开发者可以冻结你的账户（不能卖出）
    if mint.freeze_authority.is_some() {
        score -= 30;
        has_freeze_auth = true;
    }

    Ok(SecurityReport {
        is_rug_risk: has_mint_auth || has_freeze_auth,
        score,
        has_mint_authority: has_mint_auth,
        has_freeze_authority: has_freeze_auth,
    })
}
```

    // 2. 初始化 Wasm 模块 (如果是第一次)
    await init();

    // 3. 将 Uint8Array 传给 Rust 处理
    const report = analyze_mint_data(new Uint8Array(accountInfo.data));

    console.log(`Security Score: ${report.score}`);
    console.log(`Risk Message: ${report.message}`);
    
    // 4. 将结果同步给之前的 ActionCard UI
    return {
      score: report.score > 80 ? 'low' : (report.score > 40 ? 'medium' : 'high'),
      details: report.message
    };
  }
}

## 1. Chrome 插件 Wasm 运行架构

## 2. 关键配置：`manifest.json`

默认情况下，MV3 禁止执行 `eval()` 或加载远程 Wasm。你必须在 `manifest.json` 中明确放行**本地** Wasm 文件。

JSON

```
{
  "manifest_version": 3,
  "name": "Solana Intent Pilot",
  "content_security_policy": {
    "extension_pages": "script-src 'self' 'wasm-unsafe-eval'; object-src 'self'"
  },
  "web_accessible_resources": [
    {
      "resources": ["assets/*.wasm"],
      "matches": ["<all_urls>"]
    }
  ]
}
```

> **注意：** `'wasm-unsafe-eval'` 是目前加载本地 Wasm 的标准做法。

## 3. Plasmo 框架下的集成方案

既然你使用的是 **Plasmo**，它对 Wasm 有特殊的处理方式。建议将编译好的 `.wasm` 文件放在 `assets` 目录下。

### 3.1 编译 Rust

使用 `wasm-pack` 编译，目标设定为 `web`：

Bash

```
wasm-pack build --target web --out-dir ../extension/assets/wasm
```

### 3.2 动态加载逻辑 (TypeScript)

在 Plasmo 的 `sidepanel.tsx` 或 `background.ts` 中，你需要以 URL 的形式引入 Wasm：

TypeScript

```
import initWasm, { analyze_mint_data } from "url:../assets/wasm/sip_core_engine.wasm";
// 注意：url: 前缀是 Plasmo 处理静态资源的特殊语法

export const useWasm = () => {
  const runScanner = async (accountData: Uint8Array) => {
    try {
      // 初始化 Wasm 运行环境
      await initWasm(); 
      
      // 调用 Rust 函数
      const report = analyze_mint_data(accountData);
      return report;
    } catch (error) {
      console.error("Wasm Loading Error:", error);
      return null;
    }
  };

  return { runScanner };
};
```

## 4. 解决 Service Worker 中的加载问题

如果你的 Wasm 逻辑需要在 **Background Service Worker** 中运行（例如为了实现 24 小时后台监控），由于 Worker 环境没有 `fetch` 的某些默认行为，建议采用以下技巧：

TypeScript

```
// 在 background.ts 中
import wasmUrl from "url:../assets/wasm/sip_core_engine_bg.wasm";

async function loadWasmInWorker() {
  const response = await fetch(wasmUrl);
  const buffer = await response.arrayBuffer();
  // 使用 WebAssembly.instantiate 直接从 ArrayBuffer 初始化
  const { instance } = await WebAssembly.instantiate(buffer, importObject);
  return instance.exports;
}
```

## 5. 黑客松“避坑” Checklist

1. **文件大小限制：** 尽量在编译时开启优化（`opt-level = "z"`），过大的 Wasm 文件会导致插件加载变慢。
2. **异步锁：** Wasm 初始化是异步的。如果在 `init()` 完成前就调用 Rust 函数，会导致插件崩溃。务必在 UI 上做一个 `isLoadingWasm` 的状态保护。
3. **内存泄漏：** 每次调用 Wasm 都会在内存中分配空间。如果你处理高频数据流（如监控所有新交易），记得在 Rust 端实现清理逻辑，或者复用同一个 Wasm 实例。

## 2. 前端调用封装 (TypeScript)

在你的 Plasmo 插件中，你可以这样封装这个调用过程，并利用 **Zod** 库进行运行时校验（防止 LLM 抽风）：

TypeScript

```
import { z } from "zod";

// 定义严格的校验模式
const IntentSchema = z.object({
  intent: z.enum(["SWAP", "LEND", "STAKE", "TRANSFER"]),
  payload: z.object({
    inputMint: z.string(),
    outputMint: z.string(),
    amount: z.string(),
    slippageBps: z.number().default(50),
  }),
  metadata: z.object({
    confidence: z.number(),
    requires_risk_scan: z.boolean(),
  }),
});

export async function parseUserIntent(userInput: string, pageContext: string) {
  const response = await openai.chat.completions.create({
    model: "gpt-4o",
    messages: [
      { role: "system", content: SYSTEM_PROMPT_ABOVE },
      { role: "user", content: `Context: ${pageContext}\nUser Input: ${userInput}` }
    ],
    response_format: { type: "json_object" } // 强制输出 JSON
  });

  const rawJson = JSON.parse(response.choices[0].message.content);
  
  // 使用 Zod 校验数据，如果格式不对直接抛错，保护下游 Wasm 模块
  return IntentSchema.parse(rawJson);
}
```

## 3. 为什么这个 Prompt 对黑客松至关重要？

1. **确定性：** 通过 `response_format: { type: "json_object" }` 和严格的 Schema，你避免了 AI 像平常聊天那样说“好的，我为您准备好了交易指令...”这种废话。
2. **上下文感知：** 它能通过你传入的 `pageContext` 自动关联当前推文里的代币。比如用户说“买这个”，AI 能通过上下文识别出“这个”对应的 `Mint Address`。
3. **Wasm 联动：** `requires_risk_scan: true` 标志位会告诉你的插件：“嘿，现在该轮到 Rust 模块上场扫描这个 `outputMint` 了。”
