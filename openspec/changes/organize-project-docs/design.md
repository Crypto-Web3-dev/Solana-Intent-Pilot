## Context

当前 SIP 项目已有多份探索性文档，但它们集中在 `learn/` 目录中，内容风格偏 brainstorming，且同一概念在不同文件中存在重复和轻微表述漂移。为了让这些资料能服务后续实现、产品沟通与对外演示，需要建立一套正式文档目录，并约束文档的分类、命名和内容边界。

## Goals / Non-Goals

**Goals:**

- 建立一个稳定的 `docs/` 文档入口，覆盖项目总览、架构、模块、设计与路线图
- 将 `learn/` 中的内容统一为一致的术语和叙述口径
- 降低后续协作时查找资料和理解系统的成本
- 为后续文档扩展提供 OpenSpec 规范基础

**Non-Goals:**

- 不在此次变更中实现运行时代码或业务逻辑
- 不将 `learn/` 完全删除或迁移为归档系统
- 不引入额外文档站点生成器或发布流程

## Decisions

### 决策 1：建立按主题分层的 `docs/` 目录

采用 `overview/`、`architecture/`、`modules/`、`design/`、`roadmap/` 的结构，分别承载不同类型的信息，避免单文件过载。

备选方案：

- 保持单一 `项目文档.md`：简单但难以扩展，也不利于协作引用
- 完全按技术域拆分到代码目录：更贴近实现，但当前项目尚未形成稳定代码结构

### 决策 2：将 `learn/` 视为输入材料，而不是正式知识源

`learn/` 保留探索性质，`docs/` 承担正式输出职责。这样可以区分“思考过程”与“对外/对内共识文档”。

备选方案：

- 直接在 `learn/` 原地修改：成本低，但会混淆文档用途和稳定性

### 决策 3：在 OpenSpec 中为项目文档定义单独 capability

通过 `project-documentation` capability，给后续文档扩展提供统一规范和验收标准。

备选方案：

- 不建 spec，仅保留文档文件：短期可行，但后续不利于规范化演进

## Risks / Trade-offs

- [文档可能仍与未来实现脱节] → 在代码实现开始后持续回写 `docs/`
- [分类过细导致早期维护负担上升] → 当前只保留五类核心目录，避免过度设计
- [探索资料与正式资料双份维护] → 明确要求后续变更优先更新 `docs/`

## Migration Plan

1. 读取 `learn/` 中已有资料，抽取稳定内容
2. 在 `docs/` 下建立正式目录结构与索引
3. 将项目定位、架构、模块和 UI 方案写入正式文档
4. 使用 OpenSpec 记录本次文档体系化整理的 proposal、design、spec 和 tasks
5. 后续实现时将新决策优先更新到 `docs/`，必要时归档 `learn/`
