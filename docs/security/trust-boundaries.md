# SIP 信任边界与安全约束

## 1. 目标

SIP 的关键安全问题不是“有没有 AI”，而是“哪些决策可以信任 AI，哪些绝对不能只信 AI”。本文件用来明确系统中的信任边界和默认约束。

## 2. 主要边界

### 2.1 用户输入边界

来源：

- 自然语言输入
- 当前页面选中文本
- 页面中识别到的 token 文本和地址

约束：

- 所有输入都视为不可信
- 不能因为页面文本出现某个地址，就直接作为最终交易对象
- 页面检测结果只能作为候选线索，必须经过解析和校验

### 2.2 LLM 输出边界

来源：

- OpenAI / Claude / 兼容模型

约束：

- LLM 输出必须先经过 schema 校验
- LLM 不得直接触发交易
- LLM 的 `reasoning` 只能用于说明，不是安全证明

### 2.3 本地 Wasm 边界

来源：

- Rust 编译后的 Wasm 模块

约束：

- Wasm 负责可验证的静态或规则型检查
- Wasm 输出的风险报告可影响阻断逻辑
- Wasm 不直接持有钱包状态或 UI 状态

### 2.4 外部服务边界

来源：

- RPC Provider
- Jupiter API
- metadata 或 holder 数据服务

约束：

- 外部响应都可能失败、延迟或返回不完整数据
- 缺数据时不能默认当作安全
- 外部报价只能作为预览，最终提交前仍需用户确认

### 2.5 钱包签名边界

来源：

- Phantom 或兼容钱包

约束：

- 私钥与签名权完全由钱包控制
- SIP 只能请求签名，不能代持或绕过确认
- UI 必须让用户知道自己正在签什么

## 3. 默认安全规则

- 没有通过 schema 校验的 Intent 不能进入执行链
- 没有完成风险扫描的高价值交易不能进入签名链
- 风险结果缺失时应展示 `unknown`，而不是伪造 `low`
- 用户确认必须发生在最终交易内容可见之后
- 任何自动执行能力都必须显式声明并默认关闭
- MVP 默认不开放 high-risk override

## 4. 典型攻击面

### 4.1 页面投毒

攻击方式：

- 页面伪造 token 符号
- 注入误导性地址或文案

缓解：

- 地址仅作候选线索
- 必须结合 registry、上下文和风控校验

### 4.2 LLM 幻觉

攻击方式：

- 模型生成不存在或错误的 mint
- 模型误判用户想交易的对象

缓解：

- schema + 业务校验双层校验
- 低置信度进入澄清路径，不直接执行
- UI 明示解析结果

### 4.3 外部接口不稳定

攻击方式：

- RPC 限流
- quote 失败
- simulate 失败

缓解：

- 重试和备用节点
- 将失败阶段明确展示给用户
- 不把失败包装成成功或安全

### 4.4 风险误判

攻击方式：

- 数据不全导致低估风险

缓解：

- `unknown` 状态优先
- 阻断策略尽量保守
- 明确展示检查项覆盖范围

## 5. UI 安全表达要求

- 所有高风险结论都要有具体原因
- 所有阻断都要说明是哪个检查项失败
- 所有签名前页面都要展示输入资产、输出资产、协议和估算结果
- 不允许把 AI 摘要文案伪装成确定性的安全结论

## 6. 后续建议

- 增加安全事件日志
- 如未来开放高风险 override，必须设计二次确认
- 对不同协议引入更细的风险策略
- 将安全约束同步到实现测试和 QA 清单
