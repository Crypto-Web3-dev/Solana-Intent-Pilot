# SIP 风险案例库

## 1. 目标

本文件定义 MVP 阶段需要重点覆盖的风险案例，用于：

- 验证风控输出是否合理
- 验证 UI 阻断和提示是否正确
- 为 Demo 准备可控的风险路径

## 2. 案例分类

### 2.1 Mint Authority 未放弃

风险描述：

- 发行方仍可增发代币

预期风控结果：

- `level = high`
- `blocking = true`
- 明确提示 `Mint Authority Enabled`

预期 UI：

- Action Card 进入红色阻断态
- 主 CTA 禁用

### 2.2 Freeze Authority 存在

风险描述：

- 项目方可冻结账户

预期风控结果：

- 至少 `level = medium`
- MVP 中默认不单独阻断

预期 UI：

- 黄色或红色提示
- 必须展示具体原因

### 2.3 流动性过低

风险描述：

- 即便合约权限正常，也可能缺乏退出流动性

预期风控结果：

- `checks` 中包含 liquidity warning
- 若与其它风险叠加，可升级为 `high`

预期 UI：

- 明示“流动性不足”而不是仅显示总分

### 2.4 持仓集中度过高

风险描述：

- Top holders 过度集中，可能存在砸盘风险

预期风控结果：

- 通常为 `warn`
- 在 MVP 中可不默认阻断

预期 UI：

- 展示为风险细项
- 不应伪装成安全通过

### 2.5 数据缺失

风险描述：

- 无法获得流动性或 holder 数据

预期风控结果：

- 不得显示为 `low`
- 应标记为 `unknown`

预期 UI：

- 展示“数据不足，无法完整判断”

## 3. 组合风险案例

### 3.1 新币 + Mint Authority + 流动性低

预期：

- 直接阻断
- 作为高风险 Demo 样例优先保留

### 3.2 权限正常 + 流动性中等 + 持仓集中

预期：

- 允许继续，但给出 `medium`
- 用于测试黄色警示路径

## 4. 测试建议

- 每个案例准备一份模拟输入和期望输出
- 前端应验证颜色、文案、按钮状态都一致
- 风险案例尽量与真实或可复现的测试 token 对应

## 5. Demo 建议

演示时最推荐准备两条：

- 一条明显安全或低风险的成功路径
- 一条 `Mint Authority` 触发的阻断路径

这样最能体现 SIP 的“AI + 本地风控”差异化。
